## 1. 项目概述

### 1.1 项目目标

为中式八球台球桌开发一套智能AR投影系统。通过USB摄像头获取桌面视频，电脑进行视觉识别和物理计算，通过智能投影仪将进攻路线投射到桌面上，通过安卓手机APP进行控制，通过独立液晶屏显示比分和统计信息。

### 1.2 核心功能一览

| 功能 | 说明 | 状态 |
|------|------|------|
| 比赛模式 | 自动识别球色归属、推荐进攻路线、记录比分 | ✅ 已完成 |
| 训练模式 | 10档全国标准难度、50道训练题、摆球验证 | ✅ 已完成 |
| 闯关模式 | 每关连续3次成功晋级、解锁机制 | ✅ 已完成 |
| 投影路线 | 白球路径、目标球路径、杆法力度、走位落点 | ✅ 已完成 |
| 投影校准 | 投影仪与摄像头坐标对齐 | ✅ 已完成 |
| **学习引擎** | **从击球数据中学习桌物理特性，越打越准** | ✅ 已完成 |
| 语音播报 | 后端播报文字生成 + 投影仪TTS语音合成 | ✅ 已完成 |
| 杆速检测 | 摄像头检测母球初速度，投影显示实时杆速 | ✅ 已完成 |
| 进袋检测 | 连续帧对比自动检测进球事件，多帧确认防误报 | ✅ 已完成 |
| 摄像头预览 | 手机APP实时查看摄像头画面 | ✅ 已完成 |
| 投影预览 | 手机APP实时查看投影画面 | ✅ 已完成 |
| 比分网页 | 大字体比分、球状态面板、训练模式显示 | ✅ 已完成 |
| 自动发现 | UDP广播自动发现后端服务器 | ✅ 已完成 |
| 手动连接 | 手动输入IP连接（自动发现失败时备用） | ✅ 已完成 |

### 1.3 系统组成

| 组件 | 用途 | 通信方式 |
|------|------|----------|
| 💻 电脑后端 (Windows) | 视觉识别、物理计算、API服务 | — |
| 📷 USB摄像头 | 拍摄桌面，插安卓盒子，WebSocket传输视频流 | USB + WebSocket over WiFi |
| 📦 安卓盒子 (Android) | 转发USB摄像头画面到后端，安装摄像头转发APP | USB + WebSocket over WiFi |
| 📽️ 智能投影仪 (Android) | 安装投影APP，接收并显示路线 | WebSocket over WiFi |
| 📱 安卓手机 | 控制APP，模式切换/预览/校准 | REST + WebSocket over WiFi |
| 🖥️ 比分液晶屏 | NAS/软路由浏览器显示比分网页 | WebSocket over LAN |

## 2. 硬件架构

### 2.1 设备清单

| 设备 | 规格要求 | 用途 |
|------|----------|------|
| 台球桌 | 中式八球标准台 (2540×1270mm) | 打球桌面 |
| USB摄像头 | 2K@60FPS UVC免驱，2.8mm焦距，吊顶安装 | 拍摄桌面，插在安卓盒子上 |
| 智能投影仪 | 安卓系统，300+ ANSI流明，支持安装APK | 投射路线到桌面 |
| 电脑 | Windows，RX 7900 XTX，32GB RAM | 后台运算（无显示器运行） |
| 安卓手机 | 用户现有设备 | 安装控制APP |
| NAS/软路由 | 黑群晖8100 / iStoreOS J4125 | 浏览器显示比分网页 |
| 液晶屏 | VGA接口 | 显示比分 |

### 2.2 网络拓扑

```
📷 USB摄像头 ──┐
               │ USB
               ▼
         📦 安卓盒子 ──┐
                       │ WiFi 6
📽️ 智能投影仪 ────────┤
                       ├─── 🌐 路由器 ─── 💻 电脑 (Windows)
📱 安卓手机 ───────────┘                          │
                                                  │ HTTP/WebSocket
                                           ┌──────┴──────┐
                                           🖥️ NAS/软路由 ── VGA ── 🖥️ 液晶屏
```

### 2.3 坐标系统

- 以台球桌面为二维坐标系，所有坐标标准化为 `0.0 - 1.0`
- 球袋坐标：四角各一个 + 两侧长边中间各一个 = 6个球袋
- 桌球物理参数：
  - 桌岸弹性系数：0.75-0.85
  - 球与球摩擦系数：0.2
  - 球与桌面摩擦系数：0.03

### 5.3 视觉识别模块

#### 台球桌检测 (`backend/vision/table_detector.py`)

**类: `TableDetector`**

| 方法 | 说明 |
|------|------|
| `find_table()` | 三步策略：HSV颜色→HoughLines→边缘轮廓 |
| `warp()` | 应用透视变换得到桌面俯视图 |
| `transform_points()` | 坐标正/反向变换 |
| `get_table_region()` | 获取桌区域对象（角点+尺寸+变换矩阵） |

**三策略检测流程（按优先级）：**
1. **HSV 台呢颜色检测**：对绿色/蓝色台呢颜色做 HSV 范围提取→形态学闭合→最大连通区域→包围盒，首选策略
2. **HoughLines 直线估计**：Canny边缘→霍夫直线→水平/垂直聚类→中位边界估计
3. **轮廓兜底**：Canny边缘→四边形轮廓近似

**设计原因：** 原 Canny+最大轮廓方案在实景中不稳定——最大轮廓往往是灯光反射或非桌面物体。HSV 颜色定位更可靠。包裹盒直接作为桌面四角，避免凸包/多边形近似引入的畸变。

#### 球体检测 (`backend/vision/ball_detector.py` + `ball_detector_ml.py`)

**双引擎架构：ML 优先，传统 CV 兜底**

```
检测流程:
  1. ML模型已加载？→ YOLOv8 推理（主力）
  2. ML未加载/检测失败？→ 传统CV检测（兜底）
```

**传统 CV (`BallDetector`):**

| 方法 | 说明 |
|------|------|
| `detect()` | 白球(白斑检测) + 有色球(HoughCircles+颜色分类) |
| `detect_cue_ball()` | 专门检测白球 |
| `_detect_cue_by_blob()` | 白斑连通域检测（圆形度评分+邻域亮度比） |

**白球检测策略（2026-04-27 重写）：**
- 不再依赖 HoughCircles（光照暗时圆形度不足）
- 改为：HSV白色区域提取 → 轮廓查找 → 圆形度评分(4πA/P²) → 邻域亮度比(>1.25x)
- 优先选择最接近预期尺寸(12px半径)的候选

**有色球检测：**
- CLAHE 对比度增强 → HoughCircles(param2=24) → 聚类过滤 → 边缘强度验证 → 最小饱和度(45)
- 颜色分类：最近邻 HSV 颜色匹配（16 色调类型）

**已知局限：**
- 暗光环境下容易误检（台呢纹理、灯光反射）
- 建议训练 ML 模型替代

**ML 模型 (`BallDetectorML`):**

| 方法 | 说明 |
|------|------|
| `load(path)` | 加载 YOLOv8 模型 |
| `detect(warped)` | YOLOv8 推理 → Ball 对象列表 |
| `is_loaded()` | 模型是否可用 |

**模型规格：**
- 架构：YOLOv8 nano (~6MB)
- 输入：1200×600 俯视桌面图
- 输出：16 类（0=白球, 1-7=纯色, 8=黑8, 9-15=花色）
- 训练：人工标注 → `train_balls.py`

#### 进袋检测（新增模块）

**说明:** `backend/vision/pocket_detector.py` — 基于连续帧对比的进袋事件检测。

**方法:**

| 方法 | 说明 |
|------|------|
| `update()` | 传入当前帧的球列表，维护历史状态 |
| `detect_pocket_events()` | 对比上一帧与当前帧，检测进球事件 |
| `get_pocket_ball()` | 获取刚进袋的球信息 |
| `is_cue_ball_pocketed()` | 判断白球是否跟进 |

**算法策略：**

```
每帧获取球列表 → 与上帧对比：
  1. 球的ID匹配（按位置最近邻匹配）
  2. 某颗球消失 → 检查是否在袋口半径内
     ├─ 是 → 判定进袋
     └─ 否 → 可能被遮挡，标记待确认
  3. 白球消失 → 白球进袋（犯规）
  4. 进球事件 → 广播给所有客户端
```

#### 杆速检测（新增模块）

**说明:** 通过摄像头连续帧分析，测量母球被击打后的初速度，反推出杆速。

**原理:**
```
摄像头俯拍桌面(已知帧率) →
检测到母球从静止变为运动（帧N）→
跟踪母球在帧N, N+1, N+2, N+3的位置变化 →
像素位移 / 已知帧率 = 母球速度(m/s) →
母球初速度 × 质量系数 ≈ 杆速
```

**算法策略：**
1. **触发检测**：维护母球位置缓存(最近10帧)，检测到母球位置突变时触发
2. **速度计算**：取突变后连续3帧的位移平均值作为初速度
3. **噪声过滤**：排除因其他球碰撞导致的母球二次加速
4. **单位换算**：标准化坐标 → 毫米(已知桌面尺寸) → 米/秒

**数据用途:**
- 投影实时显示杆速值
- 训练模式记录每题/每档的平均杆速和标准差
- 学习引擎采集(杆速 vs 实际母球走位距离) → 优化力度预测

**特殊情况处理:**
- 多颗球同时进袋（如组合球、K球散开后入袋）
- 球被遮挡后重新出现（不是进袋）
- 帧率不稳定导致跨帧匹配失败
- 开球时球堆散开的大规模位置变化

```
current_player = 选手一（开局）
   ↓
选手击球 → 视觉检测结果 → 
  ├─ 进球（自己的球）→ current_player 不变（继续）
  ├─ 进球（自己最后一颗，打完黑8）→ current_player 获胜
  ├─ 未进球 → current_player 切换到对方
  ├─ 犯规（白球进袋等）→ current_player 切换到对方
  └─ 黑8进袋但自己球没打完 → current_player 输

**为什么这足够可靠：**
- 选手不会在轮到对方时故意去打球（违反规则）
- 手机APP和投影仪TTS都清晰显示"轮到XXX击球"
- 如果发生误判，手机上有一个"交换击球"按钮手动纠正
- 按钮放在进行中Tab的比赛界面上，不会误触

**边界情况处理：**
| 情况 | 处理方式 |
|------|----------|
| 开球后开放台（open table） | 首颗进球确定球色归属 |
| 犯规后换手 | 系统自动切换 + TTS播报原因 |
| 选手打错球（违规） | 视觉检测到不应进袋的球进袋 → 犯规处理 |
| 需要强制换手 | 手机APP "交换击球" 按钮 |

**状态机:**
```
开球(选手一)
  ↓
处理击球结果:
  ├─ 合法进球(自己的球) → 继续击球
  ├─ 合法进球(开放台)   → 分配球色，继续击球
  ├─ 未进球             → 切换到对方
  ├─ 犯规               → 切换到对方
  ├─ 犯规(白球进袋)     → 切换到对方 + 自由球
  ├─ 黑8进袋(本方球已清) → 获胜
  └─ 黑8进袋(本方球未清) → 犯规输 / 违规
```

### 5.7 学习引擎 (`backend/learning/`)

#### 设计理念

物理引擎基于理想物理定律（完美弹性碰撞、均匀摩擦力），但真实台球桌存在各种偏差：
- 库边各处弹性不同（使用久的桌岸某些位置更松）
- 台面不平（中间微凹或微凸）
- 袋口磨损导致实际袋口比标准尺寸大或小
- 台布纹理方向导致轻微偏斜

**方案：物理引擎 + AI残差修正**

```
路线预测 = PhysicsEngine.predict(初始条件) + CorrectionModel.predict(初始条件, 物理参数)
              ↑                                ↑
           理想物理定律                      数据驱动的修正
           ~1ms 确定性计算                   ~1.3万参数轻量网络
```

#### 算法流程

```
玩家击球 → 摄像头记录实际轨迹 →
  物理引擎生成预测轨迹 →
  计算残差 = 实际轨迹 - 预测轨迹 →
  残差数据 → 存入数据集 →
  后台训练AI模型 →

下一次预测:
  物理引擎生成基础预测
  AI模型预测残差修正量
  最终路线 = 基础预测 + 修正
```

#### 模块设计

**`physics_adapter.py`** — 物理参数自适应

| 方法 | 说明 |
|------|------|
| `update_from_observation(实际轨迹, 预测轨迹)` | 从单次击球中提取物理参数偏差 |
| `get_adjusted_params()` | 返回调优后的物理参数字典 |
| `reset_to_defaults()` | 重置为标准参数 |

**被调优的参数：**
| 参数 | 初始值 | 调整范围 |
|------|--------|----------|
| `CUSHION_RESTITUTION` | 0.78 | 0.65 ~ 0.90 |
| `BALL_FRICTION` | 0.03 | 0.01 ~ 0.06 |
| `POCKET_RADIUS_*` | 0.035 | 0.030 ~ 0.045 (每个袋口独立) |

**`data_collector.py`** — 击球数据采集

| 方法 | 说明 |

| `record_shot(初始状态, 物理预测轨迹, 实际轨迹)` | 记录一次击球完整数据 |
| `get_dataset()` | 获取全部已采集数据 |

    "cue_path": [[0.30,0.25], [0.50,0.25]],
    "target_path": [[0.50,0.25], [0.80,0.35], [1.0,0.5]]
  },
  "actual_observed": {
    "cue_path": [[0.30,0.25], [0.51,0.26], [0.49,0.25]],
    "target_path": [[0.50,0.25], [0.79,0.36], [1.0,0.5]],
    "target_pocketed": True,
    "cue_final_pos": [0.55, 0.30]
  },
  "residual": {
    "cue_dx": 0.01,
    "cue_dy": 0.005,
    "angle_error_deg": 1.2
  }
}
```

**`correction_model.py`** — AI修正模型

| 方法 | 说明 |
|------|------|
| `train(dataset)` | 训练/增量训练修正模型 |
| `predict(初始状态, 物理参数)` | 预测残差修正量 |
| `save(path)` | 保存模型到文件 |
| `load(path)` | 从文件加载模型 |
| `is_trained()` | 是否已有足够数据训练的模型 |

**网络结构：**
```
输入层(12维): [cue_x, cue_y, target_x, target_y, pocket_x, pocket_y,
               power, spin_x, spin_y, cushion_restitution, friction, pocket_radius]
    ↓
全连接 128 → ReLU → Dropout(0.1)
    ↓
全连接 64 → ReLU → Dropout(0.1)
    ↓
全连接 32 → ReLU
    ↓
输出层(6维): [cue_path_dx, cue_path_dy, target_path_dx, target_path_dy,
              final_pos_dx, final_pos_dy]
```

**参数量计算:**
- 12×128 + 128 + 128×64 + 64 + 64×32 + 32 + 32×6 + 6 = **12,966 参数**
- 推理时间: < 1ms (RX 7900 XTX)
- 训练时间: 500样本 ≈ 3-5秒

| `save(path)` / `load(path)` | 数据集持久化 |

#### 学习进程（玩家无感）

```
阶段 1: 冷启动 (0-50杆)
  - 仅使用物理引擎，参数为标准值
  - 系统收集击球数据，但不做修正
  - 玩家感受: 路线基本准确，但有明显误差

阶段 4: 收敛 (1000+杆)
  - 模型基本收敛，精度稳定
  - 每次击球后增量微调（不是全量重训）
  - 玩家感受: 路线可靠到可以信赖投影直接打
```

### 5.12 语音播报系统

#### 架构设计

```
后端检测到事件（进球/犯规/换手等）
  ↓
生成播报文字
  ↓
通过WebSocket发送 {"type":"announce","text":"3号球进袋"}
  ↓
投影仪APP收到消息
  ↓
Android TextToSpeech API → 扬声器播报
```

**为什么不在后端合成音频？**
- Android TTS 原生支持中文，语音自然
- 传文字比传音频带宽低得多
- 不需依赖额外TTS引擎或API

#### 后端触发逻辑

**在 `backend/game/announcer.py` 中集中管理所有播报逻辑：**

```python
class Announcer:
    """裁判播报系统——集中管理所有语音播报文字生成。"""

    def pocket_announce(self, ball_type: str, ball_color: str, is_cue: bool) -> str:
        if is_cue:
            return "犯规！白球进袋"
        name = self._ball_name(ball_type, ball_color)
        return f"{name}进袋"

    def match_start(self, player1: str, player2: str) -> str:
        return f"比赛开始，{player1}开球"

    def assign_balls(self, player: str, group: str) -> str:
        return f"{player}{group}球"

    def foul(self, reason: str) -> str:
        return f"犯规！{reason}"

    def switch_player(self, player: str) -> str:
        return f"轮到{player}击球"

    def victory(self, player: str) -> str:
        return f"本局{player}获胜！"

    def placement_feedback(self, cue_ok: bool, target_ok: bool,
                           cue_dir: str = "", target_dir: str = "") -> str:
        if cue_ok and target_ok:
            return "摆球正确"
        parts = []
        if not cue_ok:
            parts.append(f"白球{cue_dir}")
        if not target_ok:
            parts.append(f"目标球{target_dir}")
        return "，".join(parts)

    def shot_result(self, pocketed: bool, cue_in_zone: bool,
                    consecutive: int) -> str:
        if pocketed and cue_in_zone:
            msg = "目标球进袋！母球走位完美"
            if consecutive >= 3:
                msg += "，连续三连成功，进入下一题"
            return msg
        elif pocketed:
            return "目标球进袋，但母球偏离指定区域"
        else:
            return "未进球，继续努力"

    def _ball_name(self, ball_type: str, color: str) -> str:
        names = {
            "solid_yellow": "1号球", "solid_blue": "2号球",
            "solid_red": "3号球", "solid_purple": "4号球",
            "solid_orange": "5号球", "solid_green": "6号球",
            "solid_brown": "7号球",
            "stripe_yellow": "9号球", "stripe_blue": "10号球",
            "stripe_red": "11号球", "stripe_purple": "12号球",
            "stripe_orange": "13号球", "stripe_green": "14号球",
            "stripe_brown": "15号球",
            "black": "黑8",
        }
        key = f"{ball_type}_{color}" if ball_type else color
        return names.get(key, f"{color}球")
```

### 5.13 数据标注与 ML 训练管线（新增 2026-04-27）

#### 设计理念

传统 CV 球体检测在暗光环境下不可靠（台呢纹理、灯光反射产生大量误检）。采用 YOLOv8 nano 模型替代，通过人工标注→训练→部署的管线实现高精度检测。

#### 数据采集 (`backend/learning/collect_data.py`)

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `--interval` | 5.0s | 截图间隔 |
| `--max` | 0 (无限) | 最大截图数 |

**自动过滤：**
- 桌面质量检查：长宽比 1.6-2.8、亮度 >20、标准差 >10
- 去重：相邻帧像素差异 < 3% 自动跳过

#### 标注工具 (`/annotate`)

Web 端标注界面，顺序点击式工作流：

```
操作流程:
  1. 打开 http://localhost:8000/annotate
  2. 鼠标依次点击 1号→2号→...→15号→白球
  3. 不在桌面上的球按 空格 跳过
  4. 按 回车 完成标注（自动保存）
  5. ← → 切换图片
```

**标注格式（YOLO）：** `class_id x_center y_center width height`（归一化 0-1）

#### 训练 (`backend/learning/train_balls.py`)

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `--epochs` | 100 | 训练轮数 |
| `--imgsz` | 640 | 输入尺寸 |
| `--batch` | 8 | 批大小 |

## 7. 数据流设计

### 7.1 比赛模式数据流

```
摄像头帧
  ↓
视觉识别模块 → 球列表（位置+颜色+类型）
  ↓
比赛逻辑 → 当前选手→可进攻的球→目标球筛选
  ↓
┌─────────────────────────────────────────────┐
│             路线预测引擎                      │
│                                              │
│  PhysicsEngine.predict(初始条件)             │
│      ↓ (基础路线)                            │
│  + CorrectionModel.predict(初始条件, 参数)    │
│      ↓ (残差修正)                            │
│  = 最终路线                                   │
└─────────────────────────────────────────────┘
  ↓                                              ↓ (击球后)
┌──────────────┬──────────────┐          ┌──────────────┐
↓              ↓              ↓          ↓              ↓
投影渲染       API/WS         API/WS     视觉进袋检测    学习引擎
↓              ↓              ↓          ↓              ↓
投影仪 ← WiFi─ 手机 ← WiFi─  NAS ←     实际轨迹对比 → 数据采集 → 模型训练
(路线叠加)    (桌面状态)    (比分)     进球事件广播    (无感后台)
```

### 7.2 训练模式数据流

```
系统选择训练题（预定义的cue_pos, target_pos, pocket_pos）
  ↓
手机APP显示摆球要求
  ↓
选手手动摆球
  ↓
摄像头自动检测 → 验证摆球位置 → 误差允许？
  ├─ 否 → 手机显示"请调整摆球"（显示偏差方向）
  └─ 是 → 手机显示"摆球正确"
            ↓
         物理引擎计算 → 推荐路线/杆法/力度
            ↓
         投影显示路线（含走位落点标注）
            ↓
         选手击球
            ↓
         ┌──────────────────────────────────┐
         │ 视觉自动检测（连续帧对比）：       │
         │ 1. 目标球是否消失在袋口区域？      │
         │ 2. 白球最终停在什么位置？          │
         │ 3. 是否有其他球被撞入？            │
         └──────────────────────────────────┘
            ↓
         记录实际轨迹 → 对比物理引擎预测轨迹
            ↓
         残差数据 → 存入学习引擎数据集
            ↓
         判定：
         ├─ 目标球未进 → 失败 → 重新计数
         ├─ 目标球进，母球在走位区 → 成功 → 连续+1
         ├─ 目标球进，母球偏离走位区 → 进球成功但走位失败
         └─ 白球跟进 → 犯规 → 失败
            ↓
         手机APP显示结果（自动刷新，不须操作）
         投影同步更新
            ↓
         ┌─ 连续成功 ≥ 3？→ 下一题/晋级 → 自动刷新
         └─ < 3 → 重置本题 → 回到摆球步骤
```

### 7.3 校准数据流

```
手机APP点击"开始校准"
  ↓
后端：system_state["calibration"]["active"] = true
  ↓
视觉循环检测到校准模式
  ↓
渲染器生成9点十字线网格 → base64 JPEG
  ↓
WebSocket广播给投影仪 → 投影仪投射到桌面
  ↓
摄像头拍摄桌面 → 检测投影标记
  ↓
比对预期位置与实际检测位置
  ↓
手机APP轮询校准状态（每2秒）→ 显示进度
```
